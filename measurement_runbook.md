# Полная инструкция по замеру обнаружимости транспорта

> Пошаговый runbook для стенда. Держи рядом с README. Идёшь сверху вниз.
> Все запуски — на **расходном** VPS/IP, не на боевом с подпиской друзей
> (reality с донором провоцирует зондирование — можно спалить адрес).

---

## 0. Два разных замера — не путать

| | Лабораторный (Этап 4) | Полевой (Этап 4.5) |
|---|---|---|
| **Вопрос** | Насколько я *отличим* от фона? | Реально ли меня *душит* ТСПУ? |
| **Метод** | Офлайн-детектор (RandomForest) на pcap | Живой трафик через границу + замер деградации |
| **Что даёт** | TPR@FPR + топ-фичи (ЧТО палит) | Время до удушения, скорость во времени (ФАКТ) |
| **РФ-конец канала** | Не обязателен* | **Обязателен** (иначе мерим чистый канал) |
| **Расходный IP** | Желателен | **Обязателен** |

\* Лаборатория меряет *форму* трафика, а не факт блокировки. Захватываешь
туннель и фон на одной машине и спрашиваешь «различит ли классификатор» — это
валидно на любой сети. НО фон и туннель должны быть в **идентичных условиях**
(одна машина, один захватчик) — это автоматически выполняется, если снимаешь
оба класса на своём клиенте.

**Делай оба.** Лаборатория говорит, ПОЧЕМУ тебя видно; поле — душат ли НА САМОМ
ДЕЛЕ. Расхождение между ними диагностично (см. §5.5).

## 1. Что нужно до старта

### 1.1. Топология
- **Клиент** — твоя машина в Владивостоке (РФ-сторона ТСПУ).
- **Сервер** — расходный VPS **за границей** (EU/Япония/Сингапур).
- Для полевого замера трафик клиент→сервер **должен пересекать ТСПУ** — при
  заграничном сервере и РФ-клиенте это так и есть. Loopback или два заграничных
  конца = чистый канал, замер бессмысленен (это ровно твой случай «на работе всё
  работало»).

### 1.2. Расходный IP
- Отдельный дешёвый VPS, **другой провайдер/ASN**, чем боевой Xray.
- Не привязывай к нему ничего ценного (домены для переиспользования и т.п.).
- Считай, что спалишь — он по дизайну одноразовый полигон.

### 1.3. Софт (на обоих хостах)
```bash
pip install -r requirements.txt
```
На **клиенте** (где снимаешь pcap) дополнительно — **Wireshark + Npcap**
(GUI-установщик, права админа; при установке Npcap отметь loopback-поддержку,
если будешь когда-то снимать loopback).

Проверь, что захватчик виден:
```bash
python -m detect.generate --list-ifaces
```

### 1.4. Ключи и серты
```bash
# на каждом хосте:
python -m tunnel.cli keygen        # печатает private/public X25519
```
Обменяйся публичными ключами (серверу нужен свой private; клиенту — свой private
и server_public).

Для **reality/quic**: серверный self-signed cert генерится автоматически в путь
из блока `transport` конфига при первом запуске. **Скопируй `server_cert` (.crt)
с сервера на клиент** по тому же пути — клиент его пиннит.

## 2. Предполётная проверка (не пропускать)

Прежде чем снимать датасет — убедись, что всё живо. Иначе соберёшь мусор и
поймёшь это только на этапе детектора.

### 2.1. Связность и DNS
```bash
# с клиента:
dig +short <домен_или_IP_сервера>     # резолвится в IP расходного VPS
ping <IP_сервера>                      # доходит
```

### 2.2. Туннель реально несёт трафик
Подними сервер и клиент в SOCKS-режиме (см. README §5.6), потом:
```bash
curl -x socks5h://127.0.0.1:1080 https://ifconfig.me
# должен вернуть IP СЕРВЕРА (значит трафик идёт через туннель и выходит на VPS)
```
Если вернул твой домашний IP или ошибку — туннель не работает, разбирайся до
замера.

### 2.3. (Только для поля) трафик пересекает ТСПУ
Грубая проверка: `traceroute <IP_сервера>` с клиента — путь должен уходить за
границу через российские транзитные узлы. Если сервер внезапно «рядом» (РФ-ДЦ) —
ТСПУ может быть не на пути, замер деградации не покажет правды.

## 3. Лабораторный замер (Этап 4 — обнаружимость)

### 3.1. URL-лист — это negative-класс, от него зависит всё

`urls.txt` — реальные сайты, по одному URL в строке (строки с `#` игнорируются).
**Бери реалистичный микс обычного браузинга**, не один сайт:
- крупные сайты с богатым контентом (новости, соцсети, видеоплатформы);
- разные по весу страницы (лёгкие API, тяжёлые с картинками);
- HTTPS (h2/h3), как реальный браузинг.

Чем ближе список к твоему настоящему браузингу — тем честнее фон. Голый список
из 3 URL даст слабый negative-класс, и TPR будет ложно-оптимистичным.

### 3.2. Сколько собирать
- Для метрики **TPR@FPR=1e-4** нужно **≥10 000** фоновых примеров
  (`metrics.min_negatives_for_fpr`). Меньше — TPR@1e-4 упрётся в гранулярность и
  даст ложный ноль.
- Для **TPR@1e-3** хватает ~1000+, но 1e-4 — рабочая точка цензора, целься в неё.
- `--rounds` × (примеров на раунд) должно дать ≥10⁴ фона. **Подними `--rounds`
  до ~10000**, если хочешь честный 1e-4 (в README пример с 3000 — недостаточно
  для 1e-4, только для 1e-3).

### 3.3. Сбор датасета

> **Два режима — не перепутай:**
> - `--remote` (к VPS) = **ВАЛИДНЫЙ** датасет: carrier пересекает реальный NIC
>   рядом с фоном, захват на NIC видит оба класса сопоставимо.
> - `--paired` (loopback) = **только СМОУК механики**: carrier на loopback, фон
>   на NIC → на одном интерфейсе не сопоставить (туннель будет выглядеть как фон).

```bash
python -m detect.generate --list-ifaces        # номер интерфейса (бери реальный NIC)
```

**Валидный сбор — `--remote`, по одному транспорту за прогон** (на VPS поднят
сервер этого транспорта с `--exit connect` — HTTP-CONNECT выход для лаборатории):

```bash
# на VPS (свой процесс на каждый транспорт):
python -m tunnel.cli run --config config/transport.example.json --role server \
    --transport reality --exit connect

# на клиенте (РФ), захват на реальном NIC:
python -m detect.generate --remote --config config/transport.example.json \
    --transport reality --iface <NIC> --out-root data --urls urls.txt --rounds 10000 \
    --timeout 6 --max-bytes 120000
# -> data/background (прямой) + data/tunnel_reality (carrier к VPS), оба на NIC.
# Повтори для plain/padded/quic (переключив --transport здесь И на VPS).
```

Скорость (если фетчи виснут): `--timeout` — обрыв зависших (флаки-сайты типа
google.com через прокси держат коннект); `--max-bytes` — не тянуть всю страницу;
`--settle` — короче пауза старта/слива захвата. TLS-верификация в генераторе
**ВЫКЛ** (это traffic-генератор, как `curl -k`; `--verify-tls` чтобы включить).

**Локальный смоук без ТСПУ** (проверить, что конвейер жив, на работе/дома):
```bash
python tools/localsite.py --port 8443                       # терминал 1: локальный сайт
python -m detect.generate --paired --iface <LOOPBACK> --out-root data_smoke \
    --urls config/urls.local.txt --modules plain,padded,reality,quic --rounds 150
```
И фон, и туннель ходят к локальному сайту по loopback → захват на loopback-
интерфейсе видит оба класса. Меряет **carrier-vs-direct** к одному адресу; НЕ
меряет фон-реализм, V7, IP, удушение — это санити, не findings.

### 3.4. Чеклист конфаундов (проверь перед прогоном детектора)
- [ ] Фон и туннель сняты **одним** захватчиком, на **одной** машине, **одной**
      сети.
- [ ] Через туннель идёт **тот же** веб-ворклоад, что в фоне (`--remote` к VPS +
      CONNECT-выход). Никакого echo/служебного трафика в positive-классе.
- [ ] Сбор через **`--remote`** (carrier на реальном NIC), не `--paired` (loopback).
- [ ] В `urls.txt` — **сторонние** сайты, НЕ твой сервер (он не фон).
- [ ] Классы сняты **вперемежку** (короткие сессии), не большими блоками.
- [ ] Фона **≥10⁴** (если целишь в 1e-4).
- [ ] Время суток / RTT не разнесены между классами (следствие чередования).

Если хоть один пункт нарушен — детектор выучит артефакт, а не транспорт.

### 3.5. Прогон детектора
```bash
python -m detect.run_experiment --background data/background --json-out curve.json \
    plain=data/tunnel_plain padded=data/tunnel_padded \
    reality=data/tunnel_reality quic=data/tunnel_quic
```

### 3.6. Что читать в выводе
Таблица по каждому модулю:
```
режим      overhead  delay_s  TPR@1e-3  TPR@1e-4    AUC
plain        0.000    0.000     ?         ?         ?
padded       0.xxx    0.xxx     ?         ?         ?
reality      0.xxx    0.xxx     ?         ?         ?
quic         0.xxx    0.xxx     ?         ?         ?
```
+ **топ-фичи** по каждому модулю (важности RandomForest).

Как читать:
- **TPR@1e-4** — главная цифра: какая доля туннеля ловится при «цензорском»
  пороге. Чем ниже — тем лучше спрятан. `plain` ожидаемо высокий (голый Noise
  «слишком случайный»).
- **Кривая «детектируемость vs цена»**: цель — модуль в левом-нижнем углу
  (низкий TPR И низкий overhead). Тот, что прячет лучше за меньшую цену.
- **Топ-фичи** — самое ценное: они говорят, ЧТО именно палит каждый модуль.
  Дальше по §4.

## 4. Интерпретация — дерево решений (что палит → что делать)

**Не дописывай ничего, пока не прочитал топ-фичи.** Они определяют следующий шаг:

- **Топ = `v5_*` (длина ClientHello / число cipher suites / SNI)** → палит
  отпечаток хендшейка (JA3). Лечение: **faithful Reality (мини-uTLS)** —
  браузерный ClientHello вместо stdlib. Касается в основном (a).

- **Топ = `v7_unique_dests`** → палит одноэндпоинтность (весь трафик на 1 сервер
  vs веер CDN у браузинга). **mux/мультистрим НЕ лечат** — только
  **domain fronting** (много эндпоинтов). Это разворот, не доработка модуля.

- **Топ = `v7_max_concurrent_flows`** → но осторожно: надувать параллельные
  коннекты = влететь в сибирский фильтр (>3 параллельных TLS → заморозка). Не
  «чинить» в эту сторону.

- **Топ = `v2_*` (тайминги/IAT/бёрсты)** → палит временной профиль. Лечение в (c):
  **форма распределения джиттера** (равномерный сам по себе отпечаток — нужен
  ближе к естественному тяжёлохвостому).

- **Топ = `v4_*` (энтропия)** → «слишком случайный с первого байта». Это про
  `plain`; обёртки (reality/quic) должны это закрывать самим фактом TLS/QUIC.
  Если у обёртки V4 в топе — обёртка не маскирует начало потока, копай туда.

- **Топ = `v1_*` (размеры пакетов)** → padding/фрагментация в (c), подгонка под
  донор-профиль (не «размыть», а приблизить к распределению фона).

**Решение про (b)/(c) дописывать — только отсюда.** Если палит то, что эти модули
не лечат (unique_dests) — их доработка бесполезна, нужен fronting.

## 5. Полевой замер (Этап 4.5 — реальное удушение)

Параллельно лаборатории. Здесь меряешь не «похож ли», а «душат ли».

### 5.1. Долгая сессия под нагрузкой
ТСПУ душит **постепенно** — разовый «открылось» обманчив.
```bash
# через туннель (SOCKS), качай/гоняй трафик 15-20 минут, лог скорости по минутам:
while true; do
  date +%H:%M:%S
  curl -x socks5h://127.0.0.1:1080 -o /dev/null -s -w "%{speed_download}\n" \
       https://<большой_файл>
  sleep 30
done | tee field_throughput.log
```
- **Ровная скорость** → поведенческий уровень по тебе не отработал.
- **Просадка с N-й минуты / ресеты** → накопление поведенческих признаков в
  реальном времени. Запиши, на какой минуте.

Повтори для каждого транспорта (plain/padded/reality/quic) — какой держится
дольше.

### 5.2. Самозондирование (проверка уровня 3)
С **другого** адреса (не клиента) постучись на порт сервера обычным curl:
```bash
curl -v https://<IP_сервера>:<port>/
```
- Для **reality**: должен увидеть **сертификат и страницу донора** (зонд уходит
  на донор) — это правильно, уровень 3 закрыт.
- Для **plain/padded/quic**: увидит оборванное соединение / мусор — это **палево
  под активный зонд**. Зафиксируй разницу.

### 5.3. Сравнение операторов
Тот же транспорт с разных сетей: **Ростелеком дома / мобила / рабочая**. Разница
= разные профили ТСПУ (ты это уже видел на Reality). Запиши, где душит, где нет.

### 5.4. Формат лога (чтобы лёг в датасет)
На сервере веди pcap своего транспортного трафика (тот же экстрактор V1–V7), плюс
текстовый лог: timestamp старта сессии, момент первого ресета/просадки, оператор,
транспорт. Это даёт полю те же фичи, что лаборатории — для §5.5.

### 5.5. Сопоставление поле ↔ лаборатория (главная диагностика)
- **Лаборатория «чисто» + поле «душит»** → тебя палит вектор **вне твоих фич** —
  почти наверняка **IP/ASN-репутация** (§6 threat model), которую транспорт не
  лечит. Вывод: дело не в форме трафика, а в адресе → fronting / смена ASN, а не
  доработка обёртки.
- **Лаборатория «палит V5» + поле «душит»** → согласовано, лечи JA3 (faithful).
- **Лаборатория «палит» + поле «не душит»** → ТСПУ пока не отрабатывает этот
  вектор (окно затишья) — но это временно, регион догоняет.

Расхождение — это не ошибка, это самый ценный результат: оно показывает, ЧТО
именно тебя палит, точнее любого отдельного замера.

## 6. Опсек-рамки (соблюдать)
- **Только расходный IP**, не боевой с подпиской друзей.
- **Только твой трафик** на экспериментальном канале — друзей никогда.
- Самозондирование и переключение транспортов **привлекают внимание к адресу** —
  для ТСПУ это выглядит как разведка обхода. Готовься спалить полигон-IP.
- Канал включается **на замер и гасится** — не используй как запасной VPN (зря
  светит IP и размывает чистоту данных).
- Рабочий Xray на боевом VPS **не трогаешь** — он отдельно.

## 7. Траблшутинг
- **`--list-ifaces` пусто / нет dumpcap** → Wireshark+Npcap не установлен или нет
  прав админа.
- **curl через SOCKS вернул домашний IP** → сервер не в динамическом режиме
  (`target: null`) или клиент не в `--role socks`.
- **reality: клиент не коннектится** → не скопирован `server_cert` на клиент
  (клиент пиннит конкретный cert) либо `tunnel_sni` не совпадает на двух концах.
- **TPR@1e-4 = 0 подозрительно** → фона < 10⁴, метрика упёрлась в гранулярность;
  читай TPR@1e-3 или добери фон.
- **Все модули палятся одинаково высоко** → проверь §3.4: скорее всего нарушен
  конфаунд (классы сняты разными сессиями/нагрузкой), детектор учит артефакт.
- **`invalid tcpdump header` в run_experiment** → старые pcap в формате pcapng
  (dumpcap по умолчанию). capture.py теперь пишет классический pcap (`-P`);
  старые сконвертируй: `editcap -F pcap in.pcap out.pcap`, или пересними.
- **Фетчи виснут по ~20с** → флаки-сайт (google.com через прокси: HTTP/2/редиректы).
  Снизь `--timeout` (напр. 6) и убери такие сайты из `urls.txt`; `--max-bytes` тоже ускоряет.
- **`v6_duration_s` в топе фич** → вероятно артефакт латентности хопа (proxy-ness),
  а не carrier. Особенно если сервер loopback или фон/туннель к разным целям —
  сверь медианы длительности по классам.
- **Топ-фича доминирует у плохого датасета** → если фон/туннель к РАЗНЫМ адресам
  (свой сервер в фоне, или loopback-сервер при захвате на NIC) — детектор делит
  по адресу/латентности, не по обёртке. Чини топологию (§3.3), не модуль.
- **Кириллица кракозябрами** → `set PYTHONUTF8=1` (Windows).
