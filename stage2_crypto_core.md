# Этап 2 — Крипто-ядро: обоснование и минимальный туннель

> R&D-стенд. Крипто-ядро НЕ изобретаем. Обмен ключами, аутентификация сторон и
> forward secrecy — целиком за счёт аудированного Noise Protocol Framework
> (паттерн IK, реализация `noiseprotocol`). Этот документ обосновывает выбор по
> модулям и описывает запускаемый минимальный туннель point-to-point (TCP+UDP).

---

## 1. Выбор: Noise IK vs TLS 1.3 — развилка по модулям, не глобально

Как и зафиксировали в §7 Этапа 1, единого ответа нет — выбор диктуется модулем
транспорт-обёртки (Этап 3):

| Что | Ядро | Почему |
|-----|------|--------|
| Базовый туннель + модуль (b) QUIC-стиль + (c) padding/fragmentation | **Noise IK** | Не тащит X.509/ALPN/сертификатную машинерию; крипто чисто отделено от обёртки — а мы крутим именно обёртку. Маленькая, предсказуемая структура хендшейка. |
| Модуль (a) Reality-стиль | **TLS 1.3** (настоящий) | По выводу §4 Этапа 1 («The Parrot Is Dead») модуль (a) обязан быть *настоящим* TLS к живому донору, а не имитацией. Там TLS 1.3 — это и есть суть маскировки, а не деталь реализации. |

Этап 2 реализует **базовое ядро на Noise IK** (то, что переиспользуют (b)/(c)).
Настоящий TLS-стек для (a) подключается на Этапе 3 как отдельный транспорт —
его крипто тоже не пишем руками (берём реальную TLS-обвязку).

### Почему именно паттерн IK

- **Взаимная аутентификация по статическим ключам.** Обе стороны держат
  долговременные X25519-ключи; чужой не пройдёт хендшейк.
- **Сокрытие идентичности initiator.** Статический ключ клиента передаётся
  зашифрованным — пассивный наблюдатель его не видит (важно для §1.1/V5).
- **Forward secrecy** транспортных данных за счёт эфемерных ключей: компрометация
  статического ключа позже не раскрывает прошлый трафик.
- **0-RTT-нюанс IK (честно фиксируем):** первое сообщение допускает replay.
  Поэтому в msg1 мы **не передаём полезную нагрузку** — только пустой payload;
  данные идут лишь после завершения хендшейка. См. `noise_session.py`.
- **prologue** связывает хендшейк с контекстом проекта/версии
  (`transport-obfuscator/stage2/noise-ik/v1`) — защита от переигрывания
  хендшейка в другом контексте. Обе стороны обязаны иметь идентичный prologue.

Шифр-набор: `Noise_IK_25519_ChaChaPoly_SHA256` (X25519 + ChaCha20-Poly1305 +
SHA-256) — тот же класс примитивов, что в WireGuard.

## 2. TCP vs UDP: почему разные транспортные record-слои

Хендшейк идентичен (Noise IK). Разница — в том, как защищаются данные **после**
хендшейка:

- **TCP** — поток с гарантией порядка. Используем штатные `conn.encrypt/decrypt`
  noiseprotocol (неявный инкрементный счётчик-nonce). Каждое Noise-сообщение
  кадрируется 2-байтовым префиксом длины (`framing.py`).
- **UDP** — датаграммы теряются и переставляются, неявный счётчик
  рассинхронизируется. Поэтому пишем **транспортный record по модели WireGuard**
  (`record.py`): `counter(8) || AEAD`, явный счётчик-nonce + скользящее окно
  защиты от повтора (64).

### Что здесь «своё», а что аудированное (принцип №1)

`record.py` — **единственный** написанный руками крипто-смежный код. Важно, что
в нём:
- **НЕ** изобретается шифр — используется `ChaCha20Poly1305` из `cryptography`;
- **НЕ** делается обмен ключами — ключи приходят из Noise-хендшейка
  (`transport_keys()`);
- руками сделана только обвязка record: явный счётчик, раскладка nonce
  (4 нуля || counter LE — как в ChaChaPoly Noise) и replay-окно.

Это ровно то, что делает WireGuard для своего UDP-транспорта, и это не подпадает
под запрет «самопального шифра/обмена ключами». Тем не менее это место, где
ошибка дороже всего, поэтому оно покрыто тестами (roundtrip, replay, tamper).

## 3. Что реализовано (запускаемо на стенде)

```
tunnel/
  __init__.py        # константы: имя протокола Noise, prologue
  keys.py            # X25519 keygen / hex-сериализация (через cryptography)
  noise_session.py   # обёртка IK: initiator/responder, извлечение транспортных ключей
  record.py          # UDP record: counter-nonce + replay-window поверх ChaCha20-Poly1305
  framing.py         # TCP: кадрирование Noise-сообщений префиксом длины
  tcp_tunnel.py      # TCP порт-форвардер с Noise посередине (server/client)
  udp_tunnel.py      # UDP датаграммный туннель (server/client)
  cli.py             # keygen + запуск ролей по JSON-конфигу
config/
  bench.example.json # конфиг loopback-стенда (ключи демонстрационные)
tests/
  test_tunnel.py     # unit + интеграция, включая проверку «нет плейнтекста на проводе»
requirements.txt
```

Топология (оба протокола одинаковы по смыслу — порт/датаграмма-форвардер):

```
[inner app] --plaintext--> [client local_bind] --Noise IK, шифр--> [server bind] --plaintext--> [target]
```

### Запуск на стенде

```bash
pip install -r requirements.txt

# 1) сгенерировать по паре ключей на каждый хост (или взять из bench.example.json)
python -m tunnel.cli keygen

# 2) на «серверном» хосте (здесь target — то, куда туннель доставляет, напр. эхо)
python -m tunnel.cli run --config config/bench.example.json --role server --proto tcp

# 3) на «клиентском» хосте
python -m tunnel.cli run --config config/bench.example.json --role client --proto tcp
#    теперь подключения на client.local_bind (127.0.0.1:1080) уходят шифрованными
#    на сервер и форвардятся в target. Для UDP — то же с --proto udp.
```

Для двух реальных хостов разнести секции конфига: серверу нужен только свой
`static_private`; клиенту — свой `static_private` и `server_public`.

## 4. Как протестировано на обнаружимость (минимум Этапа 2)

Полноценные тесты обнаружимости (статистика потока, классификатор атакующего) —
это Этап 4. На Этапе 2 проверяем санити-минимум скрытности на уровне записи:

| Тест | Что проверяет |
|------|----------------|
| `test_record_roundtrip_and_replay` | record шифрует/расшифровывает; повтор отвергается; переупорядочивание в окне допустимо |
| `test_record_rejects_tamper` | порча AEAD-тега → отказ (целостность) |
| `test_window_not_advanced_on_tamper` | **окно сдвигается только после AEAD-верификации**: форсированный высокий counter с битым тегом не двигает окно (нет DoS на валидные пакеты) |
| `test_window_rejects_mid_window_duplicate` | повтор counter **из середины** окна режется битмаской, не только слева от окна |
| `test_record_nonce_layout_kat` | **KAT с литеральным nonce**: раскладка `4 нуля \|\| counter LE` и big-endian counter на проводе прибиты вектором (roundtrip это не ловит — он симметричен) |
| `test_noise_handshake_keys_align` | хендшейк IK сходится, транспортные ключи направлений совпадают |
| `test_tcp_tunnel_roundtrip_and_encrypted` | сквозной echo через туннель **и** через сниффер-релей: на проводе нет плейнтекста + энтропия (фича V4) высокая |
| `test_udp_tunnel_roundtrip` | сквозной UDP-echo через датаграммный туннель |

Результат: **8/8 проходит**. На TCP-проводе: плейнтекста нет, энтропия ≈7.2 бит/байт
(ожидаемо для шифртекста; это и есть baseline «голого» ядра по V4 для Этапа 4).

## 5. Компромиссы и осознанные ограничения Этапа 2

- **Отпечаток хендшейка пока «голый».** Noise-IK msg1/msg2 поверх TCP с
  префиксом длины — это узнаваемая структура (фиксированные размеры первых
  пакетов, высокая энтропия с первого байта). По логике §0/§3 Этапа 1 это
  **сам по себе признак** и именно его маскирует Этап 3 — здесь не маскируем
  намеренно, чтобы было с чем сравнивать в детекторе (baseline «голого» ядра).
- **UDP-модель минимальна:** один клиент-процесс = одна сессия; сервер — сессия
  на адрес клиента. Мультиплексирование под V7 (граф соединений) — это Этап 3,
  и оно меняет топологию, а не record.
- **Один upstream на соединение/сессию** (порт-форвардер), без полноценного
  L3/TUN — достаточно для стенда и для прогона трафика через детектор Этапа 4.
- **Счётчик-nonce не бесконечен** (2^64): для долгих сессий нужен rekey —
  отмечено в `record.py` (OverflowError), реализация rekey — за рамками Этапа 2.
  Для Этапа 4: многочасовые захваты статистики в 2^64 не упрутся (это
  ~18·10^18 датаграмм), но факт фиксируем явно, чтобы не всплыло сюрпризом —
  при сборе датасета rekey не требуется.
- **Активное зондирование / IP-репутация / корреляция** — вне scope (см. §6
  Этапа 1), ядро их не закрывает и не должно.

## 6. Выход на Этап 3

Базовое ядро (Noise IK, TCP+UDP, record-слой) готово и переиспользуемо как
транспорт под модули обёртки:
- (a) Reality-стиль — отдельный транспорт на **настоящем TLS 1.3** (не это ядро);
- (b) QUIC/HTTP3-стиль — поверх UDP-ядра;
- (c) padding/fragmentation — поверх TCP/UDP-ядра, с крутилками padding/timing.

Все три подключаются как переключаемые модули с общим интерфейсом транспорта,
после чего гоняются через детектор Этапа 4.
