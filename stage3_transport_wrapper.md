# Этап 3 — Транспорт-обёртка (переключаемые модули)

> Ядро эксперимента. Несколько подходов к маскировке как ПЕРЕКЛЮЧАЕМЫЕ модули
> поверх крипто-ядра Этапа 2, с общим интерфейсом — чтобы детектор Этапа 4 гонял
> их единообразно и сравнение было честным.

Статус: **фундамент + модули (c), (a), (b) реализованы и протестированы**
(18/18 по всему проекту). Решения по двум развилкам приняты — см. §5.

---

## 1. Общий интерфейс (критерий ревью №2)

`transport/base.py` задаёт узкий контракт `Carrier` — сообщение-ориентированный
канал для непрозрачных фреймов:

```
Carrier.send_frame(frame) / recv_frame() -> bytes|None / close()
CarrierClient.connect() -> Carrier
CarrierServer.start()/stop()  (+ on_carrier(carrier) на каждого туннель-клиента)
```

Noise-туннель Этапа 2 работает поверх ЛЮБОГО carrier одинаково
(`transport/carrier_tunnel.py`): хендшейк и перекачка данных не знают, как фреймы
выглядят на проводе. Поэтому различие модулей (a)/(b)/(c) — только в
представлении на проводе, а не в логике туннеля. Переключение модуля =
подстановка другой фабрики carrier; `_handle`/`run_initiator`/`run_responder`
не меняются. Если бы интерфейс «потёк» под специфику TLS-модуля (a), сравнение
стало бы нечестным — контракт намеренно держит (a)-специфику (стиринг на донор)
внутри модуля, за интерфейсом.

Базовый `PlainTcpCarrier` (`transport/plain_tcp.py`) — эталон «голого» ядра: по
нему детектор меряет, СКОЛЬКО даёт каждый модуль относительно немаскированного
транспорта.

## 2. Ось стоимости (критерий ревью №3)

Каждый carrier ведёт `CostStats` (`base.py`): полезные байты, байты на проводе,
логические фреймы, физические сегменты, внесённая задержка, и производное
`overhead_ratio`. Это договорённость Этапа 1 (§2/строка C + §4): без оси цены
«качество маскировки» нечитаемо — на Этапе 4 строим кривую «детектируемость vs
цена», а не оцениваем маскировку в вакууме. Каждая крутилка модуля (c) сразу
пишет свою цену (см. тесты).

---

## 3. Модуль (c): padding/fragmentation (`transport/padding.py`)

Размывает статистику размеров/таймингов (векторы V1/V2). Крутилки `PaddingPolicy`:

| Крутилка | Что делает | Цена (в CostStats) |
|----------|-----------|--------------------|
| `max_fragment_payload` | дробит логический фрейм на фрагменты | `wire_segments` (больше сегментов) |
| `min_size`/`max_size` | паддит каждый фрагмент случайными байтами до целевого размера | `wire_bytes`, `overhead_ratio` |
| `min_delay_s`/`max_delay_s` | джиттер перед отправкой фрагмента | `injected_delay_s` |
| `seed` | воспроизводимость прогона для Этапа 4 | — |

**Честное ограничение:** точная форма пакетов на проводе зависит от TCP
(сегментация/коалесценция). Ставим `TCP_NODELAY` и шлём каждый фрагмент
отдельным вызовом, но идеальный контроль размера пакета на потоке недостижим —
это аргумент в пользу датаграммных модулей (b) для строгого шейпинга.

## 5. Модули (a) и (b) — принятые решения и реализация

### (a) Reality-lite (`transport/reality.py`) — критерий ревью №1

Решение: **настоящий TLS 1.3 через stdlib `ssl` + честный relay зондов на СВОЙ
контрольный донор**, стиринг тоннель/зонд на уровне ClientHello через `MSG_PEEK`:
- SNI == секретный `tunnel_sni` → наш клиент: терминируем TLS у себя, внутри
  гоним Noise-carrier;
- иначе → зонд: прозрачно проксируем сырой TCP на донор; зонд проходит
  **настоящий TLS-хендшейк прямо с донором** и видит его сертификат и страницу.

Донор — `ControlTlsDonor`, свой реальный TLS 1.3 сервер на стенде (по твоему
указанию: без паразитной нагрузки на чужой сайт, воспроизводимо).

**Документированные fidelity-gaps** (это findings для Этапа 4, не баги):
- стиринг по **covert-SNI**, а не по аутентификатору в SessionID — stdlib `ssl`
  не даёт клиенту задать SessionID/ClientHello, а настоящий Reality прячет
  аутентификатор именно там. Фиксированный `tunnel_sni` сам по себе признак (V5).
- JA3/JA4 нашего клиента = отпечаток OpenSSL stdlib, не браузера (V5).
- Внешний TLS здесь — камуфляж; настоящая аутентификация сторон = внутренний
  Noise. Этап 4 измерит цену отсутствия uTLS.

Закрытие этих gaps = faithful Reality (мини-uTLS: кастомный ClientHello +
SessionID-auth + браузерный JA3) — отдельный объём, при необходимости отдельным
под-этапом.

### (b) QUIC/HTTP3 (`transport/quic_h3.py`)

Решение: **настоящий QUIC-стек `aioquic`, ALPN `h3`**. Честный (b) по
parrot-логике — туннель внутри реального QUIC, а не «похожие на QUIC» датаграммы
(кривой QUIC = отдельный отпечаток, §4 Этапа 1). Noise-фреймы идут
length-prefixed по одному QUIC-стриму. Мост async↔sync: фоновый asyncio-loop в
отдельном потоке; блокирующая логика туннеля — в своих потоках (иначе дедлок на
`.result()` из loop-потока).

Честные ограничения (findings для Этапа 4):
- `CostStats` на этом слое не видит QUIC-overhead (заголовки пакетов, ACK,
  паддинг до MTU) — строгий overhead для (b) считается по pcap на Этапе 4.
- **Один QUIC-стрим = нетипичный h3-профиль.** Настоящий браузерный HTTP/3
  открывает множество параллельных стримов; наш туннель сейчас — один длинный
  стрим. Parrot на уровне стека закрыт (реальный QUIC), но на уровне ПОВЕДЕНИЯ
  это отпечаток — **тот же V7 fan-out, только внутри QUIC**. Мера на Этапе 4:
  если детектор ловит (b) — проверить, не из-за одностримовости ли, а не самого
  QUIC.

### V7 (граф соединений / fan-out) — gap по ВСЕМ модулям

Зафиксировано осознанно (решение ревью Этапа 3): одиночный flow/стрим оставлен
как ограничение во всех модулях — (c)/(a) одно TCP-соединение, (b) один
QUIC-стрим — вместо веера, типичного для реального браузинга.

**Уточнение ревью Этапа 4 — V7 распадается на две разные подфичи, и «починки»
у них разные:**
- `unique_dests` — **структурно 1** для любого одноэндпоинтного туннеля (весь
  трафик на один сервер). **mux/несколько стримов это НЕ лечат** (всё равно тот
  же IP); лечится только многоэндпоинтностью (domain fronting / распределение по
  сервисам).
- `max_concurrent_flows` — можно надуть параллельными коннектами, но это бьёт в
  отдельный фильтр (>3 параллельных TLS к одному серверу → заморозка).

**Порядок measure-first:** сначала детектор Этапа 4 покажет, доминирует ли V7 и
КАКАЯ подфича; если `unique_dests` — это аргумент за разворот к fronting, а НЕ за
mux. Дорогой шаг оправдывают данные, не интуиция.

## 6. Файлы Этапа 3

```
transport/
  __init__.py
  base.py            # контракт Carrier + CostStats (ось стоимости)
  plain_tcp.py       # базовый carrier «голого» ядра (эталон)
  padding.py         # (c) padding/fragmentation/jitter + крутилки
  reality.py         # (a) Reality-lite: настоящий TLS 1.3 + relay зондов на донор
  tls_util.py        # самоподписанные cert + парсер SNI из ClientHello
  quic_h3.py         # (b) туннель внутри настоящего QUIC (aioquic, ALPN h3)
  carrier_tunnel.py  # Noise-туннель Этапа 2 поверх любого carrier
tests/
  test_transport.py  # фундамент + (c): 6/6
  test_reality.py    # (a) вкл. «зонд видит донор»: 3/3
  test_quic.py       # (b) туннель внутри QUIC: 1/1
```

| Модуль | Тест | Ключевая проверка |
|--------|------|-------------------|
| (c) | `test_transport.py` | реассембли фрагментов, overhead/джиттер в CostStats, туннель поверх padded carrier |
| (a) | `test_probe_sees_real_donor` | **зонд получает cert+страницу ДОНОРА** через TLS 1.3 (не наш сервер) |
| (a) | `test_tunnel_through_reality_tls13` | Noise-туннель внутри настоящего TLS 1.3 |
| (b) | `test_tunnel_over_quic` | Noise-туннель внутри настоящего QUIC/h3 |

Все модули включаются через ОДИН контракт (`make_server`/`CarrierClient`) —
детектор Этапа 4 гоняет их единообразно.
