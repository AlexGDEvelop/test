# Этап 4 — Лаборатория обнаружения (DPI-детектор-эмулятор)

> Без этого этапа проект бессмысленен: маскировка измеряется тем, насколько ПЛОХО
> атакующий детектор отличает наш туннель от реального фона. Метрика — TPR при
> низком FPR (рабочая точка цензора), всегда в паре со стоимостью.

Статус: **аналитический конвейер реализован и протестирован end-to-end на
синтетических pcap** (4/4; весь проект 22/22). Захват и генерация датасета —
code-complete, исполняются на стенде (нужен dumpcap/tshark + Npcap, которых в
dev-окружении нет). Реальный negative-класс (живой браузинг) — стенд-сайд.

---

## 1. Что реализовано

```
detect/
  metrics.py        # TPR@FPR (главная метрика), ROC-AUC (вспомогательно)
  features.py       # экстрактор V1–V7 из pcap (dpkt, офлайн, без Npcap)
  classifier.py     # RandomForest атакующего + важности фич + временной сплит
  cost_curve.py     # сборка «детектируемость vs цена»
  capture.py        # захват pcap ОДНИМ инструментом (dumpcap/tshark) — конфаунды
  generate.py       # размеченный датасет: туннель (plain/padded/reality/quic) + фон
  run_experiment.py # финал: прогон каждого режима через детектор -> кривая
tests/
  test_detection.py # синтетические pcap: метрика, фичи, детектор, кривая (4/4)
```

## 2. Метрика (критерий ревью №2 Этапа 4)

`metrics.tpr_at_fpr` — **TPR при FPR=1e-3 / 1e-4**, не средний ROC-AUC. AUC
считается, но как вспомогательный индикатор. `min_negatives_for_fpr` напоминает:
для измеримого FPR=1e-4 нужно ≥10 000 фоновых примеров — иначе метрика упрётся в
гранулярность и даст ложно-оптимистичный ноль.

## 3. Фичи V1–V7 (критерий №5)

`features.extract_features(pcap)` → плоский вектор:
- **V1** размеры: статистики + первые 10 размеров пакетов.
- **V2** тайминги: IAT (мс) статистики + lag-1 автокорреляция + доля бёрстов
  (помним: KS не используем как решающий, §2/V2 Этапа 1).
- **V3** up/down: доли байт/пакетов, отношение направлений.
- **V4** энтропия: первого payload и средняя. **Знак фичи нейтрален** — цель не
  «максимум энтропии», а близость к донор-профилю (§0 Этапа 1); интерпретация —
  в сравнении с фоном, а не в самой фиче.
- **V5** хендшейк (proxy): длина первого client→server payload, из ClientHello —
  число cipher suites, число расширений, длина SNI (отпечаток JA-уровня).
- **V6** длина соединения: длительность, число пакетов, число потоков.
- **V7** граф/fan-out: число уникальных адресатов, макс. одновременных потоков,
  потоков на адресата. Это та фича, что ловит одиночный flow/стрим (gap из
  Этапа 3) — `classifier.feature_importances` покажет, доминирует ли она.

## 4. Классификатор (критерий №3)

`classifier.train_eval` — RandomForest, намеренно простой (если простой ловит —
сложный тем более). **Временной сплит** (по порядку захвата) против утечки.
`feature_importances` — для measure-first решения по V7: если `v7_*` в топе, то
мультиплексирование оправдано данными; если нет — дорогую топологию не строим.

## 5. Кривая «детектируемость vs цена» (критерий №4 — финальный артефакт)

`run_experiment` для каждого модуля обучает детектор «модуль vs фон» и собирает
точку (overhead, TPR@FPR). **Overhead берётся из pcap** (суммарные байты на
сессию относительно базлайна `plain`) — это строгий pcap-overhead, обещанный для
(b), где CostStats не видит QUIC-заголовков. `cost_curve.render_table` печатает
таблицу; цель — режимы в левом-нижнем углу (низкая цена И детектируемость).

Пример вывода (на синтетике, для проверки формата):
```
режим                   overhead  delay_s  TPR@1e-3  TPR@1e-4    AUC
plain                      0.000    0.000     1.000     1.000  1.000
padded                     0.429    0.000     1.000     1.000  1.000
```

## 6. Runbook на стенде (критерий №1 — честный negative-класс)

Два уровня контроля конфаундов:
1. **Инструмент/сеть** (§3 Этапа 1): один захватчик, одна машина, одна сеть.
2. **Нагрузка** (блокер ревью Этапа 4): через туннель и напрямую тянется ТОТ ЖЕ
   реальный веб-ворклоад (те же URL). Различается только транспорт. Туннель несёт
   реальный HTTPS как CONNECT-прокси (см. `connect_proxy.py`), НЕ служебный echo —
   иначе детектор делит классы по приложению, а не по обёртке.
3. **Чередование**: `--paired` снимает в каждом раунде один URL для фона и каждого
   модуля, порядок раундов перемешан — короткие чередующиеся сессии, а не большие
   последовательные блоки (иначе детектор выучит сетевой контекст блока/время).

```bash
pip install -r requirements.txt          # + установить Wireshark (dumpcap/tshark+Npcap)

# 0) найти номер интерфейса
python -m detect.generate --list-ifaces

# 1) ПАРНЫЙ датасет: фон (URL напрямую) + туннель (тот же URL через CONNECT),
#    все модули и фон чередуются по раундам, один захватчик
python -m detect.generate --paired --iface N --out-root data --urls urls.txt \
    --modules plain,padded,reality,quic --rounds 3000
#    -> data/background, data/tunnel_plain, data/tunnel_padded, ...
#    NB: для FPR=1e-4 нужно >=10 000 фоновых примеров (metrics.min_negatives_for_fpr)
#    Золотой стандарт фона — настоящая браузерная сессия под тем же Capture;
#    --paired с реальными URL — воспроизводимый прокси этого.

# 2) ФИНАЛ: прогон каждого режима через детектор + кривая «детектируемость vs цена»
python -m detect.run_experiment --background data/background --json-out curve.json \
    plain=data/tunnel_plain padded=data/tunnel_padded \
    reality=data/tunnel_reality quic=data/tunnel_quic
```

## 7. Честные ограничения и измеряемые gaps

- **Negative-класс качества решает всё.** Прокси-фон (urllib по URL) слабее
  настоящего браузинга — для итоговых цифр снимай реальную сессию тем же Capture.
- **FPR=1e-4 требует объёма фона** (≥10⁴). На малом фоне читай TPR@1e-3 и помни
  про гранулярность.
- **V5 — proxy, не полный JA3/JA4.** Меряем длины/счётчики из ClientHello; для
  полного отпечатка — `tshark -T json` (расширение). Это и подсветит gap (a):
  covert-SNI + stdlib-JA3.
- **V7 — разделить на подфичи, иначе «V7 доминирует» прочтётся как неверная
  починка (mux):**
  - `v7_unique_dests` = **структурно 1** для любого одноэндпоинтного туннеля (весь
    трафик на сервер), у браузинга — десятки CDN. **mux/несколько стримов это НЕ
    лечат** — они всё равно на тот же IP. Лечится только многоэндпоинтностью
    (domain fronting / распределение по сервисам).
  - `v7_max_concurrent_flows` можно надуть параллельными коннектами, но это бьёт в
    отдельный фильтр (>3 параллельных TLS к одному серверу за ~60с → заморозка) —
    имитация браузерного фан-аута к СВОЕМУ серверу спалит по другому признаку.
  Вывод: если `v7_*` в топе — это аргумент за разворот к fronting, НЕ за mux.
- **Глубокие QUIC-фичи** (число стримов) — через `tshark -T json`; транспортные
  V1–V7 берутся из pcap напрямую.
- **CensorLab** (критерий №6): наш детектор намеренно простой и служит нижней
  границей. Для валидации стоит сверить хотя бы один модуль против CensorLab как
  референс-детектора, чтобы не получить ложно-оптимистичный результат от слабого
  классификатора.

## 8. Что покажет эксперимент (ожидания, не подгонка)

- `plain` — высокий TPR даже при FPR=1e-4 (голый Noise = «слишком случайный» с
  первого байта, §0 Этапа 1).
- `padded`/`reality`/`quic` — ниже TPR ценой overhead/задержки; кривая покажет,
  кто даёт лучший компромисс.
- Топ-фичи детектора скажут, ЧТО нас палит (энтропия V4? размеры V1? fan-out V7?)
  — и только это, а не интуиция, оправдывает следующий дорогой шаг.
